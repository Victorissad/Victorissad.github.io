Structur'All — SAXLogger Refactoring

Alternance · MTB111 by Creative · Septembre 2024 → en cours

Structur'All Machine® est la plateforme principale de MTB111 by Creative — une solution labellisée FinTech par Finance Innovation, utilisée par des institutions financières (BNP Paribas, Allianz, Banque de France) pour modéliser visuellement leurs processus métier sans réécrire leur SI. Ma mission : R&D sur la maintenance et l'amélioration de ce produit, en particulier le remplacement systématique d'un système de logging vétuste basé sur System.err.

Cadre
R&D Alternance · BTS SIO
Stack
Java · Spring Boot · Maven
Modules
4 modules SAX
Fichiers traités
Plusieurs centaines
Java Spring Boot Maven GitLab PowerShell IntelliJ IDEA
Mission principale

SAXLogger

Remplacement de System.err

Refactoring sur 4 modules : sax-core, sax-languages, sax-actions, sax-databases. Plusieurs centaines de fichiers Java traités.

Utilitaire 1

LoggerChecker

Validation de conformité

Outil Java développé pour scanner les fichiers du projet et vérifier qu'aucun System.err résiduel n'est présent.

Utilitaire 2

ClassOriginFinder

Analyse Reflection API

Outil d'analyse des génériques Java par Reflection. Trace l'origine des classes dans la hiérarchie de types pour faciliter le débogage.

Mission · SAXLogger

Refactoring du logging legacy

Contexte

Avant mon arrivée, l'ensemble des sorties de log de Structur'All transitait par System.err et System.out — des appels statiques non catégorisés, sans horodatage, sans niveau de criticité. Cette dette technique rendait l'application impossible à monitorer en production et bloquait toute mise en conformité avec les exigences d'auditabilité du secteur bancaire.

Stack technique

LangageJava
FrameworkSpring Boot
BuildMaven
Logger cibleSAXLogger (logger interne MTB111)
VersioningGit / GitLab — code review avec Axel Gaillard
AutomatisationScripts PowerShell de remplacement en masse

Interventions réalisées

  • Analyse préalable — recensement des occurrences de System.err / System.out sur 4 modules (sax-core, sax-languages, sax-actions, sax-databases)
  • Scripts d'automatisation — développement de scripts PowerShell de remplacement en masse pour traiter plusieurs centaines de fichiers
  • Correction d'encodage — résolution des problèmes de BOM (Byte Order Mark) UTF-8 sur les fichiers source affectés par le refactoring
  • Code review GitLab — itérations multiples avec Axel Gaillard, plus de 18 commentaires de review résolus
  • Validation manuelle — relecture des cas particuliers non automatisables, conservation des contextes spécifiques de logging
Utilitaire interne

LoggerChecker — Vérification de conformité

Objectif

Garantir qu'aucun System.err ni System.out ne soit réintroduit dans le code après le refactoring. Outil destiné à être utilisé en pré-commit ou en CI.

Fonctionnalités

  • Scan récursif d'un dossier source Java
  • Détection des appels interdits (System.err, System.out, e.printStackTrace())
  • Rapport synthétique : fichier, ligne, type d'occurrence
  • Sortie console formatée pour intégration éventuelle dans un pipeline CI
Utilitaire interne

ClassOriginFinder — Analyse Reflection

Objectif

Faciliter le débogage de Structur'All en identifiant l'origine exacte des classes Java dans la hiérarchie de types — particulièrement utile pour traiter les génériques complexes issus de l'API Reflection.

RFTG — Rental Film To Go

Projet Bloc 2 · BTS SIO SLAM · 2ᵉ année · 2025-2026

RFTG est un projet personnel autour d'un système de location de DVDs, réalisé dans le cadre du Bloc 2 de 2ᵉ année. Il regroupe quatre composants interconnectés autour d'une même API REST commune (mutualisée avec d'autres étudiants), chacun nommé d'après un personnage de l'univers Mario.

Architecture du système RFTG : les clients (web Mario, mobile Luigi) communiquent avec l'API REST Spring Boot Toad, qui accède à la base MySQL Peach via un ORM JPA.

Diagramme des cas d'utilisation côté administrateur : authentification, CRUD complet sur les films, gestion de l'inventaire.

Contexte

Application web de gestion interne pour le personnel administratif. Toutes les données sont consommées via l'API REST Toad — aucune base locale, architecture totalement découplée.

Stack technique

Interface d'administration web : connexion, liste des films (1001 entrées), gestion du stock (4581 exemplaires).

Toad

API REST centrale

Présentation

Toad est l'API REST commune au projet RFTG. Elle expose les routes de consultation du catalogue, de gestion du panier, de location, d'administration des stocks, des employés et des transactions. L'API est mutualisée entre plusieurs étudiants travaillant chacun sur leur composant.

Base de donnéesMySQL 8
BuildMaven

Architecture

L'API suit une architecture classique en couches : Controller → Service → Repository → JPA Entity. La base de données est dérivée du schéma Sakila (Oracle), enrichie de migrations versionnées (v0.2 → v0.7).

Peach

Base de données MySQL

Présentation

Peach est la base de données MySQL du projet RFTG. Le schéma est dérivé du modèle Sakila d'Oracle, adapté pour les besoins fonctionnels du système de location.

Caractéristiques

  • Schéma adapté de Sakila (modèle Oracle de référence)
  • Migrations versionnées de v0.2 à v0.7 — historique reproductible
  • Tables principales : films, dvds, clients, locations, employees, magasins
  • Contraintes d'intégrité référentielle, index sur les colonnes de recherche fréquente

Modèle conceptuel de données — entités centrales (Film, Customer, Rental, Inventory) avec leurs relations et cardinalités.

BAI Sandbox est une application Laravel 11 développée dans le cadre du Bloc 3 du BTS SIO, avec un accent fort sur la conformité réglementaire et la sécurité. Réalisation d'un audit complet selon les référentiels ANSSI, CNIL et RGPD. Bien que ce projet relève principalement du Bloc 3, il mobilise également des compétences du Bloc 1.

Sécurité

Mesures de sécurité implémentées

  • ActionLogService — journalisation complète des actions utilisateurs (qui, quoi, quand) pour traçabilité et audit
  • Rate limiting — protection des endpoints sensibles contre les attaques par force brute et le scraping
  • Conformité RGPD — cartographie des données personnelles, gestion des droits utilisateurs (accès, rectification, suppression)
  • Audit ANSSI / CNIL — rédaction d'un rapport d'audit selon les référentiels officiels
  • Validation et échappement — protection contre les injections SQL, XSS, CSRF (mécanismes natifs Laravel renforcés)

Portfolio personnel

Projet personnel · Site que vous consultez actuellement

Site web personnel réalisé pour l'épreuve E4 du BTS SIO. Conçu de bout en bout en HTML / CSS / JavaScript vanilla, sans framework ni build système, pour démontrer la maîtrise des fondamentaux du web et la capacité à structurer un projet web de A à Z. L'objectif : présenter mon parcours, mes compétences, mes réalisations et ma veille technologique de manière professionnelle et accessible.

Conception

Approche et choix techniques

Pourquoi du vanilla ?

Choix délibéré de ne pas utiliser de framework (React, Vue) ni de build system (Vite, Webpack) pour démontrer la maîtrise des fondamentaux du web. Le portfolio fonctionne en ouvrant simplement index.html, sans dépendance npm, sans serveur de build — uniquement des fichiers statiques que n'importe quel hébergement gratuit peut servir.

Architecture

  • 7 pages HTML indépendantes mais visuellement cohérentes : accueil, à propos, compétences, certifications, projets, veille, contact
  • 1 fichier CSS centralisé avec variables CSS pour le thème (light/dark), typographie modulaire et grille responsive
  • 1 fichier JS vanilla pour le toggle dark/light, le menu mobile, les animations au scroll et la persistance des préférences via localStorage
  • SVG inline pour les logos technos et les drapeaux (offline-first)
Fonctionnalités

Caractéristiques techniques

  • Mode clair / sombre avec toggle persistant (localStorage) et anti-FOUC via IIFE en début de page
  • Responsive design avec breakpoints à 800px (menu burger) et 600px (single column)
  • Accessibilité : skip link, ARIA labels, contrastes WCAG AA, navigation clavier complète
  • Performance : aucune dépendance externe sauf Google Fonts, pas de JavaScript bloquant
  • Animations subtiles au scroll via IntersectionObserver (fade-in)
  • Print-friendly : nav et footer cachés à l'impression
