Interventions réalisées

  • Scripts d'automatisation — développement de scripts PowerShell de remplacement en masse pour traiter plusieurs centaines de fichiers
  • Correction d'encodage — résolution des problèmes de BOM (Byte Order Mark) UTF-8 sur les fichiers source affectés par le refactoring
  • Code review GitLab — itérations multiples avec Axel Gaillard, plus de 18 commentaires de review résolus
  • Validation manuelle — relecture des cas particuliers non automatisables, conservation des contextes spécifiques de logging
Utilitaire interne

LoggerChecker — Vérification de conformité

Objectif

Garantir qu'aucun System.err ni System.out ne soit réintroduit dans le code après le refactoring. Outil destiné à être utilisé en pré-commit ou en CI.

Fonctionnalités

  • Scan récursif d'un dossier source Java
  • Détection des appels interdits (System.err, System.out, e.printStackTrace())
  • Rapport synthétique : fichier, ligne, type d'occurrence
  • Sortie console formatée pour intégration éventuelle dans un pipeline CI
Utilitaire interne

ClassOriginFinder — Analyse Reflection

Objectif

Faciliter le débogage de Structur'All en identifiant l'origine exacte des classes Java dans la hiérarchie de types — particulièrement utile pour traiter les génériques complexes issus de l'API Reflection.

B1.1
Gérer le patrimoine informatique
  • Le refactoring s'inscrit dans une démarche de mise en conformité avec les standards de logging professionnels (horodatage, niveaux de criticité, format structuré) attendus dans le secteur bancaire et exigés par la réglementation DORA.
  • Le système de logging structuré est une condition technique préalable à toute supervision en production. Sans logs exploitables, aucune détection d'incident ni aucun audit n'est possible.
B1.2
Répondre aux incidents et aux demandes d'assistance et d'évolution
  • Le besoin a été formulé par l'équipe interne MTB111 comme une demande d'évolution technique. J'ai pris en charge l'analyse, la conception de la solution (scripts d'automatisation) et la livraison sous validation par code review.
B1.4
Travailler en mode projet
  • Utilisation quotidienne de GitLab pour le versioning, les merge requests, les revues de code (18+ commentaires résolus avec Axel Gaillard) et le suivi des itérations sur la durée du refactoring.
  • Phase préalable d'analyse : recensement des occurrences sur 4 modules, identification des cas automatisables vs manuels, planification des itérations module par module.
B1.5
Mettre à disposition des utilisateurs un service informatique
  • Les utilitaires LoggerChecker et ClassOriginFinder ont été livrés à l'équipe MTB111 sous forme d'outils Java exécutables, accompagnés de leur documentation d'utilisation.
  • Validation systématique du refactoring par exécution des tests existants après chaque module traité, contrôle de non-régression et relecture humaine des changements automatisés.
